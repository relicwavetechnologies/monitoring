01

Problem Statement

Visa processing rules, fees, and appointment availability change with little or no notice. Currently the only way to stay current is manual, periodic checking of every relevant portal.

  • Manual effort — Staff must open each URL individually and compare against memory of what it previously said
  • Latency — A policy change published on a Monday morning may not be noticed until Friday, causing incorrect advice to clients in the interim
  • Scale — Five URLs is the seed; the real list grows. Manual checking does not scale
  • No audit trail — There is currently no record of when a change occurred or what exactly changed, only that something is different
  • Noise fatigue — Any naive alerting system (e.g. cron + curl + diff) would produce hundreds of false positives per day from CDN timestamps, cookie banners, and A/B test variants
Goal

A single dashboard that shows all five sites in one place, highlights meaningful changes with AI-generated summaries, maintains a full history of every change detected, and sends a well-formatted email when something actually matters — with zero false positives surviving into your inbox.

Monitored Sites

Each site requires a hand-tuned adapter that knows where meaningful content lives, what render mode is needed, and how frequently to poll. The table below is the seed configuration.

URL Authority Render Poll Noise Risk
in.usembassy.gov/mumbai/ US Embassy Mumbai — visa news, fee tables, appointment notices Static Daily Low
visa.vfsglobal.com/are/en/sgp/apply-visa VFS Global UAE→Singapore — appointment slots, document requirements JS Daily High
www.gov.uk/standard-visitor/… UK Home Office — Standard Visitor Visa requirements and fees Static Daily Low
www.vfsglobal.com/en/individuals/index.html VFS Global global landing — announcement banner, country portal links JS Daily High
www.ica.gov.sg/ Singapore ICA — pass and visa announcements, news feed Static Daily Medium
VFS noise note

VFS Global sites are notoriously noisy — session tokens, real-time slot counters, CDN variant headers, and regional A/B tests all mutate the DOM on every render. The adapter selector must target the announcement region only — not the booking widget — and strip patterns must be aggressive. The two-poll stability window that applies to all sites is especially important for both VFS adapters.

03

Detection Pipeline

Every poll traverses the same stages. The LLM is the last gate, not the first — by the time a diff reaches it, all obvious noise has been discarded.

Full pipeline · one tick per site per poll cycle
01
Scheduler
Vercel Cron fires once daily at 09:00 UTC (0 9 * * *). The tick endpoint queries the DB for sites due for a poll, then fans each out as a separate background function — so per-site failures are isolated.
02
Fetcher
renderMode: 'static' → native fetch() for speed. renderMode: 'js' → headless Playwright to fully execute JavaScript before extracting HTML.
03
Scope + Strip
Apply the adapter's contentSelector to isolate the meaningful region. Run all stripPatterns to remove session tokens, timestamps, counters, and cookie banners. Normalise whitespace.
04
Hash Check — Filter 1
SHA-256 the cleaned text and compare to the last stored snapshot hash. Identical → stop here. No diff, no LLM, no write. Eliminates the vast majority of polls on quiet days.
05
Min-diff Threshold — Filter 2
Generate a character-level text diff. If total added + removed chars < minDiffChars (default 40), treat as noise and stop. Kills micro-edits, whitespace drift, single-character corrections.
06
Stability Window — Filter 3
The diff hash is stored in Site.pendingDiff (a JSON field). The change must match on two consecutive polls before escalating to the LLM. If the hash shifts again before confirmation it is reset, not escalated. Eliminates CDN flicker, A/B test variants, and transient cache misses.
07
LLM Classification
gpt-5.4-mini receives the site label, URL, and extracted text diff (not raw HTML). Returns structured JSON: category, severity 1–5, summary, confidence 0–1. Served via a custom OpenAI-compatible gateway. Called on <5% of all polls.

04

Site Adapters

Each site has a small typed adapter object. The adapter is the only place that knows site-specific details — the pipeline itself is generic.

adapter interface TypeScript 
interface SiteAdapter {
  id:              string           // unique slug, e.g. "vfs-are-sgp"
  url:             string           // canonical URL to fetch
  name:            string           // human-readable name for UI
  renderMode:      'STATIC' | 'JS'  // fetch() vs Playwright
  contentSelector: string           // CSS selector of meaningful region
  stripPatterns:   string[]         // regex sources stored in Postgres array
  pollIntervalMin: number           // cadence hint (cron currently daily)
  // stability window (2 polls) and minDiffChars (40) are global constants,
  // not per-adapter — they live in the pipeline code, not here.
}
One-time AI bootstrap

When onboarding a new site, we feed the fully-rendered HTML to gpt-5.4 once. It identifies the CSS selector covering policy / fee / appointment content, and suggests up to 8 strip-pattern regexes for noise (session IDs, timestamps, counters). The result is reviewed, saved to the database as the site adapter, and never re-generated unless the site redesigns. AI bootstraps the config once; all subsequent monitoring is deterministic.

05

Filter Layers

Three deterministic filters run before any LLM call. Together they eliminate >95% of all diffs, keeping OpenAI token spend near zero for quiet periods.

Layer 1
Scope + Strip
Apply the adapter's CSS selector to extract only the meaningful content region. Run all strip regexes to remove session tokens, CSRF fields, timestamps of the form HH:MM:SS, view counters, and cookie-consent markup. Normalise all whitespace. Hash the result (SHA-256). If hash matches the previous snapshot — stop, no diff.
Kills header / footer / sidebar / ad churn
Layer 2
Min-diff threshold
Compute a character-level diff of the cleaned text (not raw HTML). If the total added + removed character count is below MIN_DIFF_CHARS = 40 (a global constant in the pipeline), treat as noise and stop. This kills single-character typo corrections, whitespace-only changes, and tiny widget state mutations.
Kills micro-edits, whitespace drift
Layer 3
Stability window
The diff hash is stored in the Site.pendingDiff JSON column. On the next poll, if the same content hash is seen again, the diff is promoted to LLM classification. If the page changed again before confirmation, pendingDiff is updated and the clock resets. Always requires exactly two consecutive matching polls — no per-site override. Transient CDN variants, server-side A/B tests, and stale cache flushes all disappear before the next poll.
Kills flicker, CDN variants, A/B tests

06

AI Classification

Only diffs that survive all three filter layers reach the LLM. The model's job is to classify the change and summarise it — not to decide whether the page changed.

Model
gpt-5.4-mini
Default for all classification. Fast, structured output via a custom OpenAI-compatible gateway. Falls back to gpt-5.4 for confidence < 0.6 cases requiring a second pass.
Output format
Structured JSON
category, severity 1–5, one-line summary, confidence 0–1. Parsed directly into Postgres. No free-form text in the critical path.
Trigger threshold
sev ≥ 3 + conf ≥ 0.7
Both conditions must be true for an email to fire. Ambiguous low-confidence hits are stored on the dashboard for manual review.
Categories
7 change types
POLICY_CHANGE · FEE_CHANGE · APPOINTMENT · DOCUMENT_REQUIREMENT · NAVIGATION · COSMETIC · UNKNOWN. Each maps to a label and severity colour in the dashboard.
Cost discipline
< 5% of polls
Three pre-AI layers mean the LLM is called on fewer than 1 in 20 polls on quiet sites. At $0.00015/1K tokens, even heavy usage stays under $2/month.
Prompt context
Diff + site context
The prompt includes the site label, URL, and the extracted text diff — not the raw HTML. Keeps token count low and reduces hallucination risk.
LLM response schema TypeScript 
interface ClassificationResult {
  category:   'POLICY_CHANGE' | 'FEE_CHANGE' | 'APPOINTMENT' |
              'DOCUMENT_REQUIREMENT' | 'NAVIGATION' |
              'COSMETIC' | 'UNKNOWN'
  severity:   1 | 2 | 3 | 4 | 5   // 1 = cosmetic, 5 = critical
  summary:    string                  // one line, ≤ 140 chars
  detail:     string                  // extended explanation, ≤ 600 chars
  confidence: number                  // 0.0 – 1.0
}
07

Dashboard

Built in Next.js 15 App Router with React Server Components. Two views: the feed and the per-site detail. Protected by NextAuth v5 magic-link (Resend) so only authorised users can access the dashboard.

Overview feed

  • Stats bar — total sites, total changes, notable changes (severity ≥ 3), and active site count — all rendered server-side on every load
  • Significant changes — reverse-chronological list of all changes with severity ≥ 3, one-line AI summary, category label, and severity colour strip
  • Minor / cosmetic feed — lower-severity changes shown below in the same editorial list style
  • Send Digest button — in Settings; triggers an immediate Resend email with the top 10 notable changes to the authenticated user's address

Per-site detail view

  • Change history — full timeline of every diff detected, cosmetic or otherwise, with timestamp
  • Diff viewer — inline coloured text diff (green = added, red = removed) of the extracted content region before and after the change
  • Snapshot count — total snapshots taken shown in the stats bar; raw snapshots stored as gzipped bytes in Postgres
  • Adapter config — render mode, content selector, strip patterns, and poll interval visible in a collapsible details panel on each site page

08

Email Alerts

Sent via Resend. Emails are built as plain HTML with inline styles — Instrument Serif headings, IBM Plex Mono labels, and the same #6C63FF accent palette as the dashboard. Two modes: automatic per-event alerts and an on-demand digest.

Email structure
  • Subject — [Important] VFS Global UAE→SGP: Application fee raised to $X effective May 1 — format: [Severity label] Site name: AI summary (≤ 80 chars)
  • Hero card — severity + category pill badges, site name, detection timestamp
  • AI summary — one-line and extended explanation written by the LLM, not a raw diff
  • Change block — highlighted added text (green) and removed text (red), up to 20 diff lines
  • Action link — "View full diff →" deep-links into the dashboard change detail page
  • Dashboard CTA — footer section with "Open Dashboard →" button linking to the overview
Two delivery modes
  • Automatic alert — fires immediately when a change clears the severity ≥ 3 + confidence ≥ 0.7 gate. One email per qualifying event.
  • On-demand digest — "Send Digest" button in the Settings page emails the top 10 notable changes (severity ≥ 3) to the authenticated user on demand. Useful for a morning briefing or sharing with colleagues.
Severity 1–2 events

Minor and cosmetic changes are stored in the database and visible on the dashboard for manual review — they never trigger an automatic email. The on-demand digest also filters to severity ≥ 3 only.

09

Tech Stack

Chosen for minimal moving parts while keeping the scraping worker fully separate from the Next.js frontend — Vercel functions are not suitable for long-running Playwright sessions.

Frontend
Next.js 15 · App Router
Dashboard UI rendered as React Server Components. API routes handle poll triggers, digest dispatch, and site management. Hosted on Vercel Hobby.
Scheduler
Vercel Cron
Single daily cron at 09:00 UTC hits a tick endpoint, which fans out per-site poll functions as isolated background tasks. Hobby plan supports one cron entry.
Scraping
fetch + Playwright
Static sites use native undici/fetch for speed. JS-rendered sites (VFS) use Playwright with @sparticuz/chromium bundled into the Vercel function.
Database
Postgres (Neon)
Change events, site config, and full page snapshots (gzipped HTML + text stored as bytea columns). Stability window tracked via Site.pendingDiff JSON field. Neon serverless autoscaling.
ORM
Prisma v7 + Neon adapter
Type-safe queries using the @prisma/adapter-neon edge-compatible adapter. Migrations managed via prisma migrate dev against the Neon direct connection.
AI
gpt-5.4-mini / gpt-5.4
Classification with structured JSON output via a custom OpenAI-compatible gateway. gpt-5.4-mini is the default; gpt-5.4 is the low-confidence fallback. One-time adapter bootstrap also uses gpt-5.4.
Auth
NextAuth v5 (Auth.js)
Magic-link email sign-in via Resend. JWT strategy with Prisma adapter. Middleware guards the entire dashboard route group.
Email
Resend
Handles both auth magic links and change-alert emails. HTML built with inline styles matching the dashboard design system. Free tier sends from onboarding@resend.dev.

Rollout Plan

Three phases, roughly two weeks total. The goal is a working alert email by end of Phase 1 — the dashboard and polish follow.

Phase 1 · Days 1–5
Pipeline first, UI second
  • Neon DB schema + Prisma models
  • Static fetcher + SHA-256 hash check
  • Two adapters: gov.uk + ICA (static, low noise)
  • LLM classify + Postgres write
  • First working alert email via Resend
Phase 2 · Days 6–10
JS sites + stability
  • Playwright fetcher for VFS adapters
  • 2-poll stability window via pendingDiff
  • Vercel Cron tick + per-site fan-out
  • gzipped snapshot storage in Postgres
  • NextAuth magic-link auth guard
Phase 3 · Days 11–14
Dashboard + polish
  • Next.js dashboard — overview + site detail
  • Change detail page with diff viewer
  • On-demand digest button in Settings
  • Adapter bootstrap wizard (AI-generated config)
  • Email template design polish + deep links
Adapter tuning period

After Phase 1 goes live, run for 48 hours in log-only mode — all diffs stored, no emails sent. Review the log to see what the filters are passing or blocking, then tune selectors and strip patterns before turning alerts on. This is especially important for the two VFS adapters.

Current Status — Prototype / Demo

The version running today is a working prototype. The full detection pipeline, dashboard, and email delivery are implemented and functional on Vercel Hobby. The cron schedule is daily (Vercel Hobby limitation); per-site polling intervals and Playwright-based JS rendering are staged for Phase 2. Site adapters are seeded with realistic configuration but may need selector and strip-pattern tuning once live data starts flowing.
