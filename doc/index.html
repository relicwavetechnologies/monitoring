01

Problem Statement

Visa processing rules, fees, and appointment availability change with little or no notice. Currently the only way to stay current is manual, periodic checking of every relevant portal.

  • Manual effort — Staff must open each URL individually and compare against memory of what it previously said
  • Latency — A policy change published on a Monday morning may not be noticed until Friday, causing incorrect advice to clients in the interim
  • Scale — Five URLs is the seed; the real list grows. Manual checking does not scale
  • No audit trail — There is currently no record of when a change occurred or what exactly changed, only that something is different
  • Noise fatigue — Any naive alerting system (e.g. cron + curl + diff) would produce hundreds of false positives per day from CDN timestamps, cookie banners, and A/B test variants
Goal

A single dashboard that shows all five sites in one place, highlights meaningful changes with AI-generated summaries, maintains a full history of every change detected, and sends a well-formatted email when something actually matters — with zero false positives surviving into your inbox.

Monitored Sites

Each site requires a hand-tuned adapter that knows where meaningful content lives, what render mode is needed, and how frequently to poll. The table below is the seed configuration.

URL Authority Render Poll Noise Risk
in.usembassy.gov/mumbai/ US Embassy Mumbai — visa news, fee tables, appointment notices Static 30 min Low
visa.vfsglobal.com/are/en/sgp/apply-visa VFS Global UAE→Singapore — appointment slots, document requirements JS 15 min High
www.gov.uk/standard-visitor/… UK Home Office — Standard Visitor Visa requirements and fees Static 1 hr Low
www.vfsglobal.com/en/individuals/index.html VFS Global global landing — announcement banner, country portal links JS 15 min High
www.ica.gov.sg/ Singapore ICA — pass and visa announcements, news feed Static 30 min Medium
VFS noise note

VFS Global sites are notoriously noisy — session tokens, real-time slot counters, CDN variant headers, and regional A/B tests all mutate the DOM on every render. The adapter selector must target the announcement region only — not the booking widget — and strip patterns must be aggressive. A two-poll stability window is mandatory for both VFS adapters before any diff is escalated.

03

Detection Pipeline

Every poll traverses the same stages. The LLM is the last gate, not the first — by the time a diff reaches it, all obvious noise has been discarded.

Full pipeline · one tick per site per poll cycle
01
Scheduler
Inngest cron fires at the interval defined per adapter — 15 min for VFS, 30 min for embassies, 1 hr for gov.uk.
02
Fetcher
renderMode: 'static' → native fetch() for speed. renderMode: 'js' → headless Playwright to fully execute JavaScript before extracting HTML.
03
Scope + Strip
Apply the adapter's contentSelector to isolate the meaningful region. Run all stripPatterns to remove session tokens, timestamps, counters, and cookie banners. Normalise whitespace.
04
Hash Check — Filter 1
SHA-256 the cleaned text and compare to the last stored snapshot hash. Identical → stop here. No diff, no LLM, no write. Eliminates the vast majority of polls on quiet days.
05
Min-diff Threshold — Filter 2
Generate a character-level text diff. If total added + removed chars < minDiffChars (default 40), treat as noise and stop. Kills micro-edits, whitespace drift, single-character corrections.
06
Stability Window — Filter 3
Record the diff in pending_changes with a pass counter. The change must survive stabilityPasses consecutive polls (default 2, VFS = 3) before escalating. Eliminates CDN flicker, A/B test variants, and transient cache misses.
07
LLM Classification
GPT-4o-mini receives the site label, URL, and extracted text diff (not raw HTML). Returns structured JSON: category, severity 1–5, summary, confidence 0–1. Called on <5% of all polls.

04

Site Adapters

Each site has a small typed adapter object. The adapter is the only place that knows site-specific details — the pipeline itself is generic.

adapter interface TypeScript 
interface SiteAdapter {
  id:              string           // unique slug, e.g. "vfs-are-sgp"
  url:             string           // canonical URL to fetch
  label:           string           // human-readable name for UI
  renderMode:      'static' | 'js'  // fetch() vs Playwright
  contentSelector: string           // CSS selector of meaningful region
  stripPatterns:   RegExp[]         // noise patterns to remove before hashing
  pollIntervalMin: number           // cron cadence in minutes
  stabilityPasses: number           // consecutive polls change must survive
  minDiffChars:    number           // ignore diffs smaller than this
}
One-time AI bootstrap

When onboarding a new site, we feed the fully-rendered HTML to GPT-4o once with the prompt "Where on this page would visa policy changes, fee changes, or appointment availability changes appear? Return a CSS selector for that region and a list of regex patterns to strip before change-detection (session IDs, timestamps, counters)." The output is reviewed, saved as the adapter, and never re-generated unless the site redesigns. AI bootstraps the config once; all subsequent monitoring is deterministic.

05

Filter Layers

Three deterministic filters run before any LLM call. Together they eliminate >95% of all diffs, keeping OpenAI token spend near zero for quiet periods.

Layer 1
Scope + Strip
Apply the adapter's CSS selector to extract only the meaningful content region. Run all strip regexes to remove session tokens, CSRF fields, timestamps of the form HH:MM:SS, view counters, and cookie-consent markup. Normalise all whitespace. Hash the result (SHA-256). If hash matches the previous snapshot — stop, no diff.
Kills header / footer / sidebar / ad churn
Layer 2
Min-diff threshold
Compute a character-level diff of the cleaned text (not raw HTML). If the total added + removed character count is below the adapter's minDiffChars setting (default 40), treat as noise and stop. This kills single-character typo corrections, whitespace-only changes, and tiny widget state mutations.
Kills micro-edits, whitespace drift
Layer 3
Stability window
Record the diff in a pending_changes table with a passes counter. On the next poll, if the same diff is still present, increment. Only when passes ≥ stabilityPasses (default 2, set to 3 for VFS) does the diff graduate to the LLM stage. Transient CDN variants, server-side A/B tests, and stale cache flushes all disappear before the next poll.
Kills flicker, CDN variants, A/B tests

06

AI Classification

Only diffs that survive all three filter layers reach the LLM. The model's job is to classify the change and summarise it — not to decide whether the page changed.

Model
GPT-4o-mini
Default for all classification. Cheap, fast, structured output. Reserve GPT-4o for confidence < 0.6 cases requiring a second pass.
Output format
Structured JSON
category, severity 1–5, one-line summary, confidence 0–1. Parsed directly into Postgres. No free-form text in the critical path.
Trigger threshold
sev ≥ 3 + conf ≥ 0.7
Both conditions must be true for an email to fire. Ambiguous low-confidence hits are stored on the dashboard for manual review.
Categories
5 change types
policy_change · fee_change · appointment_availability · cosmetic · navigation. Each maps to a colour and icon in the dashboard.
Cost discipline
< 5% of polls
Three pre-AI layers mean the LLM is called on fewer than 1 in 20 polls on quiet sites. At $0.00015/1K tokens, even heavy usage stays under $2/month.
Prompt context
Diff + site context
The prompt includes the site label, URL, and the extracted text diff — not the raw HTML. Keeps token count low and reduces hallucination risk.
LLM response schema TypeScript 
interface ClassificationResult {
  category:   'policy_change' | 'fee_change' |
              'appointment_availability' | 'cosmetic' |
              'navigation' | 'unknown'
  severity:   1 | 2 | 3 | 4 | 5   // 1 = cosmetic, 5 = critical
  summary:    string                  // one line, ≤ 120 chars
  confidence: number                  // 0.0 – 1.0
}
07

Dashboard

Built in Next.js with real-time updates via Server-Sent Events. Two views: the feed and the per-site detail. No auth complexity needed for a single-user internal tool — protected by a simple token-based route guard.

Feed view

  • Status row per site — site name, last checked time, last change detected, current severity badge
  • Alert stream — reverse-chronological list of all changes severity ≥ 3, with AI summary and diff preview
  • Severity filter — filter feed by category or severity; all stored even if not alerted
  • Live indicator — SSE-connected; new alerts appear without page refresh

Per-site detail view

  • Change history — full timeline of every diff detected, cosmetic or otherwise, with timestamp
  • Diff viewer — side-by-side or inline text diff of the content region before and after
  • Snapshot access — link to the full stored HTML snapshot at any point in time (R2/S3)
  • Adapter config — editable selector and strip patterns inline so you can tune without code deploys

08

Email Alerts

Sent via Resend. Each email is a React Email template rendered server-side at send time. One email per alert event — no digest batching, since visa changes are time-sensitive.

Email structure
  • Subject — [VisaWatch] fee_change · VFS UAE→SGP · severity 4
  • Site block — full name, URL, authority, timestamp of detection
  • AI summary — one sentence written by the LLM, not a raw diff
  • Change block — highlighted added text (green) and removed text (red), limited to the changed paragraphs
  • Action link — deep link into the dashboard diff view for this exact change event
  • Metadata footer — severity, confidence score, change category, poll cycle number
Why not a daily digest?

A daily digest trades immediacy for inbox cleanliness. For visa policy monitoring, a fee change published at 9 AM that you only see at 5 PM may mean a full day of incorrect advice to clients. Individual emails for severity ≥ 3 events preserves the latency advantage of near-real-time monitoring. Severity 1–2 events are only visible on the dashboard — they never email.

09

Tech Stack

Chosen for minimal moving parts while keeping the scraping worker fully separate from the Next.js frontend — Vercel functions are not suitable for long-running Playwright sessions.

Frontend
Next.js 15 · App Router
Dashboard UI, SSE endpoint for live feed updates, React Email template rendering. Hosted on Vercel.
Worker
Inngest
Durable cron-based job execution with built-in retries, step-level observability, and a generous free tier. Handles all scraping and pipeline jobs outside Vercel's 60 s function limit.
Scraping
fetch + Playwright
Static sites use native fetch for speed. JS-rendered sites (VFS) use Playwright with a headless Chromium instance managed in the Inngest worker.
Database
Postgres (Neon)
Change events, adapter config, snapshots metadata, and pending_changes stability tracking. Neon offers serverless autoscaling and a free tier sufficient for this workload.
Blob storage
Cloudflare R2
Full HTML snapshots stored as objects keyed by siteId + timestamp. R2 has no egress fees. 30-day retention policy via lifecycle rule.
AI
OpenAI GPT-4o-mini
Change classification with structured JSON output. GPT-4o as fallback for low-confidence cases. Key held server-side in environment variables only.
Discovery
Serper API
Secondary signal only — daily Google SERP query for each authority domain to catch announcements published on pages not in the monitored URL list.
Email
Resend + React Email
Clean Next.js-native email infrastructure. React Email gives fully typed, testable email templates. Resend handles deliverability.
ORM
Prisma
Type-safe database access shared between the Next.js app and the Inngest worker via a shared package in the monorepo.

Rollout Plan

Three phases, roughly two weeks total. The goal is a working alert email by end of Phase 1 — the dashboard and polish follow.

Phase 1 · Days 1–5
Pipeline first, UI second
  • Neon DB schema + Prisma models
  • Static fetcher + SHA-256 hash check
  • Two adapters: gov.uk + ICA (static, low noise)
  • LLM classify + Postgres write
  • First working alert email via Resend
Phase 2 · Days 6–10
JS sites + stability
  • Playwright worker for VFS adapters
  • Stability window (pending_changes table)
  • R2 snapshot storage
  • Inngest cron jobs per adapter
  • Serper daily discovery scan
Phase 3 · Days 11–14
Dashboard + polish
  • Next.js dashboard — feed + detail views
  • SSE live update endpoint
  • Diff viewer component
  • Adapter config editor UI
  • Email template polish + deep links
Adapter tuning period

After Phase 1 goes live, run for 48 hours in log-only mode — all diffs stored, no emails sent. Review the log to see what the filters are passing or blocking, then tune selectors and strip patterns before turning alerts on. This is especially important for the two VFS adapters.
